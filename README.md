# CodeVector Products

A full-stack product browsing app with cursor-based (keyset) pagination, built for 200k+ products.

## Stack

- **Backend:** Node.js, Express, Mongoose, MongoDB
- **Frontend:** React (Vite)

## Quick Start

### Prerequisites

- Node.js 18+
- A MongoDB Atlas cluster (or local MongoDB instance)

### 1. Clone and install

```bash
cd codevector-products

# Backend
cd backend
cp .env.example .env   # then fill in MONGODB_URI and PORT
npm install

# Frontend
cd ../frontend
npm install
```

### 2. Configure environment

Edit `backend/.env`:

```
MONGODB_URI=mongodb+srv://<user>:<password>@cluster0.xxxxx.mongodb.net/codevector-products
PORT=3001
```

### 3. Seed the database

```bash
cd backend
node seed.js
```

This inserts 200,000 products (configurable with `--count=N`).  
Each product has a `created_at` rounded to the nearest minute, so many products
share the exact same timestamp — this intentionally stresses the pagination
tie-breaking logic.

### 4. Start the backend

```bash
cd backend
npm start
```

### 5. Start the frontend

```bash
cd frontend
npm run dev
```

Open http://localhost:5173. The Vite dev server proxies `/api` requests to the
backend.

## Pagination Design

### Why keyset/cursor pagination instead of offset?

Two hard requirements drove this choice:

**1. Performance at scale**

`skip()`/`offset()` pagination must scan and discard all preceding rows on
every request. On page 5000 of 200k rows, the database still reads 100k+ rows
just to skip them. Keyset pagination uses the index to jump directly to the
correct position — the query cost is **O(log n)** per page regardless of depth.

**2. Consistency under concurrent writes**

Offset pagination breaks when rows are inserted or removed mid-session:

- **Duplicates:** If 50 new products are inserted at the top (newest), every
  existing row shifts forward. A user on page 2 suddenly sees the row that was
  at the bottom of page 1 again.
- **Skips:** Similarly, rows can jump past the page boundary unread.

Cursor pagination pins the position with `(created_at, _id)`. New rows sort
*after* the cursor (they have a higher created_at or, at the same millisecond,
a higher _id), so the current session never sees them. The feed is
**snapshot-consistent** for that session.

### How it works

```
Sort:  created_at DESC, _id DESC
Page:  WHERE (created_at < cursor.createdAt)
          OR (created_at == cursor.createdAt AND _id < cursor.id)
```

- `created_at` is the primary sort key (newest first).
- `_id` is the tie-breaker. Since `_id` is monotonically increasing and
  generated by MongoDB, it provides a strict total order. Even if 500 products
  share the exact same `created_at`, the cursor is unambiguous.

The cursor is an opaque base64-encoded JSON blob sent back by the client on
the next request. The client never inspects it.

## Testing Pagination Correctness with `simulate-writes.js`

This script simulates background write activity while you browse:

```bash
# Run once
node simulate-writes.js

# Run continuously (every 3 seconds) — recommended for testing
node simulate-writes.js --loop
```

It does two things each cycle:

1. **Inserts 50 new products** with `created_at = now` (appear at the top of
   the newest-first feed).
2. **Updates 50 random products** (changes price, bumps `updated_at`, but
   leaves `created_at` untouched — position in the feed is preserved).

### How to verify no duplicates / no skips

1. Open the frontend (http://localhost:5173).
2. In a second terminal, run `node simulate-writes.js --loop`.
3. Paginate through products by clicking "Load More".
4. Verify:
   - **No duplicates:** No product appears on two different pages.
   - **No skips:** You don't suddenly jump past products you haven't seen.
   - **Updates are invisible:** The feed order is unchanged (only `price` and
     `updated_at` change, which aren't sort keys).
   - **New inserts are invisible:** New products have a newer `created_at`, so
     they sort after your cursor — they won't appear until you refresh.

## Indexes

### `{ category: 1, created_at: -1, _id: -1 }`

Used when filtering by a specific category. The query uses the index as
follows:

1. **`category: 1`** — Equality filter narrows to one category.
2. **`created_at: -1`** — Sorts newest-first within that category.
3. **`_id: -1`** — Tie-breaks when multiple products share the same
   `created_at`.

Without `_id` in the index, MongoDB would need to do an in-memory sort of
documents with the same `created_at`, which is both slower and could produce
inconsistent ordering across queries.

### `{ created_at: -1, _id: -1 }`

Used when no category filter is applied. Same reasoning — `created_at` for the
primary sort, `_id` for tie-breaking.

Both indexes are **covered** by the cursor pagination query, meaning all the
data MongoDB needs to sort and filter is right in the index — no random page
lookups for sort.

## API Endpoints

| Method | Path              | Description                         |
|--------|-------------------|-------------------------------------|
| GET    | `/api/health`     | `{ status: "ok" }`                  |
| GET    | `/api/categories` | `{ categories: [...] }`             |
| GET    | `/api/products`   | Cursor-paginated product listing    |

| Param      | Default | Description                        |
|------------|---------|------------------------------------|
| `limit`    | 20      | Items per page (max 100)           |
| `category` | —       | Exact category filter              |
| `cursor`   | —       | Opaque cursor from previous page   |
